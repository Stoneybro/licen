# Data Lifecycle

Five phases. No single point of trust. Here's every step a dataset takes from upload to model delivery.

---

## Phase 1: Publish

**Actor:** Dataset Owner (browser)

1. Publisher selects a JSONL file
2. Browser generates a random **AES-256-GCM key**
3. Dataset is encrypted client-side → encrypted blob
4. AES key is ECIES-sealed with the Orchestrator's public key → **key envelope**
5. Encrypted blob + envelope uploaded to **0G Storage** → `datasetRoot` (Merkle root) returned
6. Policy manifest JSON uploaded to 0G Storage → `manifestHash` returned
7. Publisher signs a transaction calling `registerDataset(datasetRoot, manifestHash, policyParams)` on the **DataPolicy contract**

**State after phase:** Dataset is on-chain. Policy is enforced. Plaintext has never touched LICEN's servers.

---

## Phase 2: Discovery

**Actor:** Envio Indexer + Marketplace UI

1. Envio detects `DatasetRegistered(datasetRoot, owner, manifestHash)` event
2. Indexes the event into GraphQL
3. Marketplace UI hydrates dataset cards from Envio GraphQL

**State after phase:** Dataset is visible to researchers in the marketplace, with live on-chain policy data.

---

## Phase 3: Access Request

**Actor:** AI Researcher

1. Researcher selects dataset, configures epoch count
2. UI calculates `escrow = royaltyPerEpoch × requestedEpochs`
3. Researcher signs `requestAccess(datasetRoot, purposeId, epochs)` — escrow is locked
4. DataPolicy contract validates against policy (epoch caps, session TTL, allowed purposes)
5. Contract emits `AccessGranted(jobId, datasetRoot, requester, epochs)`
6. Envio indexes the event

**State after phase:** Escrow is locked. Job is `Granted`. Researcher cannot access data yet — the Orchestrator hasn't acted.

---

## Phase 4: Orchestration

**Actor:** Orchestrator (background Node.js process)

1. Orchestrator polls Envio every 5 seconds for `Granted` jobs
2. For each new job:
   - Verifies on-chain state is `Granted` (contract is ground truth, not Envio)
   - Fetches sealed key envelope from Web App API (`/api/orchestrator/key-envelope?datasetRoot=...`)
   - Decrypts envelope with ECIES private key → **AES key in memory**
   - Downloads encrypted dataset from 0G Storage
   - Decrypts blob in memory → plaintext JSONL
   - Re-uploads plaintext to 0G Storage → `plaintextRootHash`
   - Deletes temp file; zeros AES key bytes
   - Calls `@0gfoundation/0g-compute-ts-sdk` → `createFineTuningTask(provider, model, plaintextRootHash, epochs)` → `taskId`
   - Saves `taskId` + `providerAddress` to `compute_jobs` DB table
   - Calls `startJob(jobId)` on DataPolicy contract → state moves to `Running`

**State after phase:** Job is `Running`. AES key is gone from memory. Plaintext is temporarily on 0G Storage (required by current 0G Compute SDK). Training is dispatched.

---

## Phase 5: Training & Settlement

**Actor:** 0G Compute Provider + Orchestrator

1. GPU provider downloads plaintext dataset from 0G Storage
2. Fine-tuning runs inside **TEE enclave** (hardware-isolated)
3. LoRA adapter uploaded to 0G Storage by provider → `resultHash`
4. Orchestrator's `jobTracker` polls task status every 30 seconds

When status = `Delivered`:
- Orchestrator calls `acknowledgeModel(taskId, provider)` — provider fee settlement

When status = `Finished`:
- Orchestrator calls `confirmTrainingComplete(jobId, actualEpochs, resultHash, attestationRef)`
- DataPolicy contract:
  - Releases `actualEpochs × royaltyPerEpoch` to the publisher
  - Refunds unused escrow `(requestedEpochs − actualEpochs) × royaltyPerEpoch` to researcher
  - Sets job state to `Completed`

5. Researcher sees job as `Completed` in their session view
6. Researcher downloads LoRA adapter using `resultHash`

**State after phase:** Publisher paid. Researcher has their model. Settlement is on-chain and auditable.

---

## Failure Handling

If any step in Phase 4 or 5 fails:

| Failure point | What happens |
|---|---|
| Key decryption error | Orchestrator calls `markJobFailed()` → researcher auto-refunded |
| 0G Storage download error | Same — `markJobFailed()` + refund |
| Compute dispatch error | Same — `markJobFailed()` + refund |
| 0G Compute task `Failed` | `jobTracker` calls `markJobFailed()` + refund |
| Orchestrator crash/restart | `jobTracker` resumes all `running` jobs from DB on startup |

The researcher is never stuck waiting — failed jobs always trigger a refund path.
