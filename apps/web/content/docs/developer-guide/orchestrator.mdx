# The Orchestrator

The Orchestrator (`packages/orchestrator`) is a critical security and operational component of the LICEN platform. It runs as a continuous Node.js background process.

## Core Responsibilities

1.  **Polling:** Continuously queries the Envio Indexer for new `AccessGranted` events.
2.  **Key Management:** Holds the private key necessary to decrypt the AES Key Envelopes generated by the Web App.
3.  **Data Transformation:** Downloads encrypted datasets from 0G Storage, decrypts them in memory, and re-uploads them in plaintext for compute providers.
4.  **Compute Dispatch:** Uses the `@0gfoundation/0g-compute-ts-sdk` to send fine-tuning tasks to GPU providers.
5.  **State Tracking:** Updates a local PostgreSQL database (`compute_jobs` table) to track the lifecycle of long-running tasks.

## Security Considerations

The Orchestrator is the *only* service in the LICEN stack that has access to plaintext dataset data (temporarily in memory) and the private key required to decrypt user envelopes.

*   **Isolation:** It must be deployed in a secure environment without public SSH access.
*   **API Secret:** It communicates with the Next.js Web API using a shared password (`ORCHESTRATOR_API_SECRET`). If this secret is compromised, an attacker could fetch envelopes, but without the Orchestrator's private key, they still cannot decrypt the data.

## The Dispatch Workflow

When a new job is found, `dispatcher.ts` handles the heavy lifting:

1.  **Fetch Envelope:** Requests the envelope from `https://your-web-app.com/api/orchestrator/key-envelope`.
2.  **Unseal:** Decrypts the envelope using `eciesjs`.
3.  **Download:** Fetches the encrypted blob from 0G Storage using the `ZgFile` SDK.
4.  **Decrypt:** Uses Node's built-in `crypto` module (AES-256-GCM) to decrypt the blob in memory.
5.  **Re-upload:** Uploads the resulting `JSONL` file back to 0G Storage and gets a new `plaintextRootHash`.
6.  **Dispatch:** Calls `createFineTuningTask` via the 0G Compute SDK, providing the `plaintextRootHash`.

## Environment Variables

For the Orchestrator to function, it needs three distinct private keys:
*   `ORCHESTRATOR_PRIVATE_KEY`: Secp256k1 key for decrypting ECIES envelopes.
*   `BACKEND_WALLET_PRIVATE_KEY`: EVM wallet with testnet tokens to pay for 0G Storage uploads.
*   `OG_COMPUTE_PRIVATE_KEY`: EVM wallet with testnet tokens to pay for 0G Compute tasks.
